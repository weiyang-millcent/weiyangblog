# CTF Misc 与流量分析知识点汇总

> **博主寄语**：在 Misc（杂项）题型中，工具的熟练度与知识点的积累至关重要。本篇博客记录了在参加软件安全赛中所遇到的核心考点，涵盖隐写术处理逻辑与流量分析实战。

---

##  Part I: CTF 隐写术

在 Misc 题目中，解题过程往往环环相扣。从处理加密压缩包到识别并解码不可见字符，通常遵循以下逻辑：

### 1. 压缩包密码破解策略

当面对加密压缩包且无直接密码时，按序尝试以下方案：

* **密码碎片搜集与拼接**
    * **来源**：检查前序关卡中的图片备注、十六进制文件尾（Strings 提取）或隐藏的文本片段。
    * **操作**：将多个字符串按逻辑顺序排列组合，拼接成最终密码。
* **CRC32 爆破**
    * **适用场景**：压缩包内文件极小（通常 $\le 4$ 字节）时。
    * **原理**：利用 CRC32 值公开的特性，通过穷举字符组合，计算并对比 CRC32 值。若匹配，则可直接还原内容。
* **伪加密识别与修复**
    * **特征**：压缩包提示密码，但十六进制编辑器中全局方式标记位（Flags）异常。
    * **修复**：修改 ZIP 结构的标志位（如将 `09` 改回 `00`），即可消除加密状态。

---

### 2. 零宽字节隐写 

若解压后得到肉眼观察为“空白”的 `txt` 文件，通常涉及零宽字符隐写。

#### A. 隐写特征识别
零宽字符在普通编辑器中不显示，但在字节流中真实存在。使用 Linux 命令 `hexdump -C` 可使其“显形”：



```bash
# 典型的零宽字符 Hex 编码 (UTF-8)
e2 80 8b  # Zero Width Space (ZWSP)
e2 80 8c  # Zero Width Non-Joiner (ZWNJ)
---


### B. 解码还原逻辑

零宽隐写工具的核心是将**二进制位 (Bits)** 映射到不可见字符上。解码通常遵循以下三个标准步骤：

1.  **特征提取**：遍历文件字节流，过滤掉所有可见的 ASCII 字符，仅保留特定的零宽字符序列（如 `e2 80 8b`）。
2.  **二进制转换**：根据题目设定的映射规则进行还原。
    > **常见映射示例**：
    > * `e2 80 8b` (ZWSP) $\rightarrow$ `0`
    > * `e2 80 8c` (ZWNJ) $\rightarrow$ `1`
3.  **ASCII 还原**：将每 8 位二进制（1 Byte）组合，对照 ASCII 编码表转换回可见字符，最终拼接出 **Flag**。

##  Part II: CTF 流量分析 (Traffic Analysis)

### 流量安全分析：Apache Shiro 反序列化漏洞复盘

#### 0x01 核心原理：硬编码密钥与序列化
Apache Shiro 1.2.4 及更低版本的核心问题在于其默认使用了**硬编码的 AES 密钥**（例如：`kPH+bIxk5D2deZiIxcaaaA==`）来处理 `RememberMe` 功能。



* **攻击链条**：
    `构造恶意 Java 对象` $\rightarrow$ `序列化` $\rightarrow$ `AES 加密` $\rightarrow$ `Base64 编码` $\rightarrow$ `填入 Cookie`
* **脆弱点**：由于密钥公开，攻击者可以伪造合法的 Cookie。服务端接收后执行 `readObject()`，导致 **RCE (远程代码执行)**。

#### 0x02 流量分析指纹 (Pcap Analysis)
在进行流量回溯时，主要通过以下关键特征判定攻击行为：

* **关键字段识别**：
    * **Cookie 异常**：请求头中包含长度极其巨大的 `rememberMe` 字段。
    * **特征响应**：响应头中出现 `Set-Cookie: rememberMe=deleteMe`。
        * *注：这通常代表 Key 匹配成功，但 Payload 解析失败或正在进行密钥爆破。*
* **载荷特征 (Payload Data)**：
    解密后的数据流如果包含 Java 序列化魔数 `0xAC ED 00 05`，即可确认是针对 Shiro 的反序列化尝试。

#### 0x03 恶意 Webshell 的提取与还原
由于现代攻击多采用**内存马**（无文件攻击），必须从流量中提取：

1.  **流量解密**：使用流量分析脚本加载对应密钥，从 `.pcap` 包中还原原始序列化字节流。
2.  **字节码剥离**：从序列化流中定位敏感类，提取出封装在其中的 `.class` 字节码文件。
3.  **代码还原**：利用反编译工具（如 JD-GUI, Jadx）分析提取出的字节码，从而掌握攻击者的 C2 通信协议与后门逻辑。


##  总结与避坑指南

在处理**零宽隐写**题目时，有一个致命陷阱：**绝对不要直接从普通剪贴板中复制粘贴文本内容**。

* **原因**：许多编辑器、浏览器或在线剪贴板会自动执行“格式化”，将不可见字符视为冗余噪声并直接剔除。一旦字符丢失，隐写信息将彻底损坏。
* **建议**：始终操作原始文件的 **字节流 (Byte Stream)**，或在支持显示 Unicode 原始编码的专业编辑器中操作。
