# 用kali虚拟机解决镜像文件分析题目

这道题是需要对img缓存文件进行分析，对其进行文件的复原提取以及分析过程。此前我一直用WSL进行这类操作，此次我第一次使用kali虚拟机进行复现解题，借此题记录相关知识并熟悉对kali的使用。

### 分析文件构成

我们要对一个img文件进行分析，首先在kali终端执行这两条指令：
`fsstat ~/Desktop/case.img
fls -r -p ~/Desktop/case.img`

``

发现了以下的相关文件：

`d/d 11: lost+found`

`d/d 12: etc`

`d/d 13: etc/edgebox`

`r/r 21: etc/edgebox/data_policy.conf`

`d/d 14: var`

`d/d 15: var/log`

`d/d 16: var/log/edgebox`

`r/r 22: var/log/edgebox/runtime_20260421.log`

`d/d 17: var/lib`

`d/d 18: var/lib/edgebox`

`d/d 19: var/lib/edgebox/audio`

`r/r 23: var/lib/edgebox/audio/audio_cache.db`

`r/r 24: var/lib/edgebox/audio/audio_cache.db-wal`

`r/r 25: var/lib/edgebox/audio/audio_cache.db-shm`

`d/d 20: var/lib/edgebox/uplink`

`V/V 16385: $OrphanFiles`

关键文件如下：

inode 21 ：etc/edgebox/data_policy.conf

inode 22 ：var/log/edgebox/runtime_20260421.log

inode 23 ：var/lib/edgebox/audio/audio_cache.db

inode 24 ：var/lib/edgebox/audio/audio_cache.db-wal

inode 25 ：var/lib/edgebox/audio/audio_cache.db-shm

### 提取配置文件

输入icat ~/Desktop/case.img 21 > config.conf，得到config.conf文件，其内容是：

`audio_retention=metadata_only
audio_cache_cleanup=true
uplink_policy=headquarter_v3`

`udio_cache_cleanup=true` 表示**应当自动清理缓存**。

既然清理开关为开，却仍有缓存文件，说明清理流程未正常执行。

### 提取运行日志

输入icat ~/Desktop/case.img 22 > runtime.log，获得runtime.log日志文件，其内容为：

`audio=A20260421-A03-042 cleanup=success
audio=A20260421-A03-088 qa_cache=metrics_only
audio=A20260421-A03-117 cleanup=success sqlite_state=wal_pending
audio=A20260421-A03-139 cleanup=success probe=old`

日志声称 `cleanup=success`，但 `sqlite_state=wal_pending` 表明 **WAL 中仍有挂起数据**。

这直接证明了清理并不彻底，WAL 未真正提交或删除。

``

### 📂 第一类：文件定位与确认（准备工作）

| 指令                                              | 作用                                                                         |
| ----------------------------------------------- | -------------------------------------------------------------------------- |
| `sudo find / -name "*.img" -type f 2>/dev/null` | 在 Kali 全系统中搜索 `.img` 文件，找到镜像的实际位置（因为最开始 `~/case.img` 路径不存在）                |
| `ls -lh ~/Desktop/case.img`                     | 确认镜像文件是否存在于桌面，并查看其大小是否正常                                                   |
| `mmls ~/Desktop/case.img`                       | 查看镜像的分区表。这里执行后**无输出**，说明镜像是**单个 ext4 裸分区**（无 MBR/GPT），后续所有命令无需加 `-o` 偏移量参数 |

* * *

### 🧬 第二类：文件系统元数据分析（核心取证）

| 指令                                   | 作用                                                                                                    |
| ------------------------------------ | ----------------------------------------------------------------------------------------------------- |
| `fsstat ~/Desktop/case.img`          | **查看文件系统超级块信息**：卷名（`EDGEBOXDATA`）、挂载点（`/tmp/cxw_store_audio_wal_case/mnt`）、是否干净卸载、inode 数量等           |
| `fls -r -p ~/Desktop/case.img`       | **递归列出根目录下所有现存文件/目录的完整路径**，并显示 inode 编号。通过此命令发现了 `/var/lib/edgebox/audio/` 目录下的三个关键文件（inode 23、24、25） |
| `fls -r -p ~/Desktop/case.img 16385` | 尝试查看 `$OrphanFiles` 虚拟目录（未输出，属于正常现象）                                                                  |

* * *

### ⛏️ 第三类：按 inode 提取文件内容（icat）

| 指令                                                     | 作用                                                                                             |
| ------------------------------------------------------ | ---------------------------------------------------------------------------------------------- |
| `icat ~/Desktop/case.img 21 > config.conf`             | 提取 inode 21（`etc/edgebox/data_policy.conf`）到本地文件，查看缓存清理策略                                      |
| `icat ~/Desktop/case.img 22 > runtime.log`             | 提取 inode 22（`var/log/edgebox/runtime_20260421.log`）到本地文件，查看运行日志中的 `cleanup` 和 `wal_pending` 状态 |
| `icat ~/Desktop/case.img 23 > /tmp/audio_cache.db`     | 提取 inode 23（SQLite 主库）到 `/tmp` 目录                                                              |
| `icat ~/Desktop/case.img 24 > /tmp/audio_cache.db-wal` | 提取 inode 24（SQLite WAL 预写日志）到 `/tmp` 目录                                                        |
| `icat ~/Desktop/case.img 25 > /tmp/audio_cache.db-shm` | 提取 inode 25（SQLite 共享内存文件）到 `/tmp` 目录                                                          |

* * *

### 📊 第四类：查看配置文件与日志内容

| 指令                                         | 作用                                                           |
| ------------------------------------------ | ------------------------------------------------------------ |
| `cat config.conf`                          | 查看配置文件内容，发现 `audio_cache_cleanup=true`（明确要求清理）               |
| `tail -20 runtime.log` 或 `cat runtime.log` | 查看运行日志，定位到关键行 `cleanup=success` 但 `sqlite_state=wal_pending` |

* * *

### 🗄️ 第五类：SQLite 数据库查询

| 指令                                                         | 作用                                                |
| ---------------------------------------------------------- | ------------------------------------------------- |
| `sqlite3 /tmp/audio_cache.db ".tables"`                    | 查看主库中有哪些表（可选）                                     |
| `sqlite3 /tmp/audio_cache.db 'select * from audio_cache;'` | 查询 `audio_cache` 表，发现主库中只有 042、088、139，**没有** 117 |

* * *

### 🔎 第六类：二进制/文本搜索与上下文提取

| 指令                                                                    | 作用                                                               |
| --------------------------------------------------------------------- | ---------------------------------------------------------------- |
| `strings -a /tmp/audio_cache.db-wal \| grep 'A20260421-A03-117'`      | 从 WAL 文件中提取可读字符串并搜索目标音频 ID，输出 `53A20260421-A03-117`，确认存在         |
| `strings -a /tmp/audio_cache.db-wal \| grep -C10 'A20260421-A03-117'` | 搜索该 ID 并显示**前后 10 行上下文**，成功捕获了完整的 base64 编码缓存记录（`eyJ...J9`）及其时间戳 |

> 💡 **关于 `strings` 参数**：`-a` 表示扫描整个文件而不止是数据段，`-C10` 显示匹配行前后各 10 行内容。

* * *

### 🧩 第七类：Base64 解码验证（Python）

| 指令                                                                                                                                         | 作用                                                                                                |
| ------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------- |
| `python3 -c "import base64, json; s = '...'; obj = json.loads(base64.b64decode(s)); print(json.dumps(obj, indent=2, ensure_ascii=False))"` | 将从 WAL 中截获的 base64 字符串解码为 JSON，最终得到包含 `audio_id: A20260421-A03-117`、`purge_mode: wal_only` 等完整元数据 |

* * *

### 📎 辅助指令（用于排查问题）

| 指令                                                                       | 作用                                                                    |
| ------------------------------------------------------------------------ | --------------------------------------------------------------------- |
| `ls -lh /tmp/audio_cache.*`                                              | 确认提取的三个 SQLite 文件已成功生成并查看大小（主库 4K、WAL 21K、SHM 32K）                    |
| `strings -a /tmp/audio_cache.db-wal \| grep -E 'eyJ.*A20260421-A03-117'` | 尝试用正则匹配完整 base64 行（此处无输出，说明该行并非以 `eyJ` 开头且包含该 ID 的完整格式，改用 `-C10` 才成功） |
